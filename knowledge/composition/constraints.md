# Composition Constraints — What the UI Auto-Does That You Must Do Manually

## Why This File Exists

When a user drags-and-drops in the Studio UI, many intermediate nodes, bindings, and transformations are auto-generated by internal plugins (paste worker, DnD handler, normalization). **The agent writes raw JSON — it doesn't drag-and-drop.** So the agent must explicitly create everything the UI would auto-generate, or the composition will be broken.

**READ THIS BEFORE GENERATING ANY COMPOSITION JSON.**

---

## 1. Slot UIDs Must Be Explicit

When the UI drops a component, it auto-generates a unique `slotUid` for each slot prop and creates the `slots` object.

**The agent must do this manually:**

```json
{
  "uid": "my-section",
  "type": "section",
  "props": {
    "children": { "type": "slot", "slot": "slot-abc123" }
  },
  "slots": {
    "slot-abc123": [
      { "uid": "child-1", "type": "header", "props": { "..." : "..." } }
    ]
  }
}
```

**Rules:**
- Every slot prop MUST have a `"slot"` field pointing to a unique slot UID
- The `slots` object MUST have a key matching that UID
- The slot UID must be unique across the entire composition
- If a component has multiple slots (e.g., Layout with header/sidebar/main), each needs its own UID

**Common mistake:** Using `"children"` as the slot key directly. The slot key is a **generated UID**, not the prop name. The prop name is in `props`, the slot UID is in `props.*.slot` and `slots.*`.

---

## 2. Condition Blocks for Modular Block Repeaters

When the UI binds a repeater to a modular blocks field, it auto-inserts a **condition-block** node inside the repeater for each block type. This is because modular blocks can have different schemas per block type, so each type needs its own template.

**The agent must create condition blocks manually:**

```json
{
  "uid": "sections-repeater",
  "type": "repeater",
  "props": {
    "items": {
      "type": "array",
      "value": [],
      "binding": {
        "type": "template",
        "value": { "path": { "sections": {} } }
      }
    }
  },
  "slots": {
    "slot-rep-123": [
      {
        "uid": "cb-hero-section",
        "type": "condition-block",
        "metadata": {
          "condition": {
            "type": "modular_block",
            "operator": "eq",
            "value": "hero_section",
            "conditionBinding": {
              "type": "repeater",
              "value": {
                "repeaterUID": "sections-repeater",
                "path": { "_content_type_uid": {} }
              }
            },
            "dataBinding": {
              "type": "repeater",
              "value": {
                "repeaterUID": "sections-repeater",
                "path": { "hero_section": {} }
              }
            }
          }
        },
        "props": {
          "children": { "type": "slot", "slot": "slot-cb-hero" }
        },
        "slots": {
          "slot-cb-hero": [
            {
              "uid": "cb-hero-box",
              "type": "box",
              "props": {
                "children": { "type": "slot", "slot": "slot-box-hero" }
              },
              "slots": {
                "slot-box-hero": [
                  {
                    "uid": "hero-heading",
                    "type": "header",
                    "props": {
                      "text": {
                        "type": "string",
                        "value": "",
                        "binding": {
                          "type": "repeater",
                          "value": {
                            "repeaterUID": "sections-repeater",
                            "path": { "hero_section.heading": {} }
                          }
                        }
                      }
                    }
                  }
                ]
              }
            }
          ]
        }
      },
      {
        "uid": "cb-content-section",
        "type": "condition-block",
        "metadata": {
          "condition": {
            "type": "modular_block",
            "operator": "eq",
            "value": "content_section",
            "conditionBinding": {
              "type": "repeater",
              "value": {
                "repeaterUID": "sections-repeater",
                "path": { "_content_type_uid": {} }
              }
            },
            "dataBinding": {
              "type": "repeater",
              "value": {
                "repeaterUID": "sections-repeater",
                "path": { "content_section": {} }
              }
            }
          }
        },
        "props": {
          "children": { "type": "slot", "slot": "slot-cb-content" }
        },
        "slots": {
          "slot-cb-content": [
            {
              "uid": "cb-content-box",
              "type": "box",
              "props": {
                "children": { "type": "slot", "slot": "slot-box-content" }
              },
              "slots": {
                "slot-box-content": [
                  /* content section template components here */
                ]
              }
            }
          ]
        }
      }
    ]
  }
}
```

**Key rules:**
- One `condition-block` per modular block type
- `condition.type` = `"modular_block"`
- `condition.operator` = `"eq"`
- `condition.value` = the block type UID (e.g., `"hero_section"`)
- `conditionBinding` binds to `_content_type_uid` field via repeater context
- `dataBinding` binds to the block type UID field via repeater context
- **Each condition-block must contain a `box` node** — condition blocks auto-insert boxes
- Child components inside the box bind via repeater with path prefixed by block type: `"hero_section.heading"`

## 3. Condition Blocks for Reference Repeaters

Similar to modular blocks, when a repeater iterates over a reference field that can reference multiple content types, condition blocks are needed per content type.

```json
{
  "uid": "cb-author-type",
  "type": "condition-block",
  "metadata": {
    "condition": {
      "type": "reference",
      "operator": "eq",
      "value": "author",
      "conditionBinding": {
        "type": "repeater",
        "value": {
          "repeaterUID": "related-repeater",
          "path": { "_content_type_uid": {} }
        }
      }
    }
  }
}
```

## 4. Box Wrappers for Repeaters

The UI auto-wraps repeaters in `box` containers when:
- Repeater is dropped as a sibling (not into an empty slot)
- Repeater is dropped into another repeater
- Target has multiple slots
- Target slots already have children

**The agent must wrap repeaters in boxes when appropriate:**

```json
{
  "uid": "repeater-wrapper-box",
  "type": "box",
  "metadata": { "repeaterWrapper": true },
  "props": {
    "children": { "type": "slot", "slot": "slot-box-wrap" }
  },
  "slots": {
    "slot-box-wrap": [
      {
        "uid": "my-repeater",
        "type": "repeater",
        "props": { "items": { "..." : "..." } },
        "slots": { "..." : [] }
      }
    ]
  }
}
```

**Rules:**
- Mark the wrapper box with `metadata.repeaterWrapper: true`
- A repeater inside another repeater ALWAYS needs a box wrapper

## 5. Static Value Bindings (Paste Worker)

When the UI inserts a node via drag-and-drop, the **paste worker** transforms every prop:
1. Generates a new UID for the node
2. For each prop, creates a `static_value` binding: `{ type: "static_value", value: "uid-propName" }`
3. Extracts the prop's actual value into the `staticValues` store

**The agent's vibeChangesApplier handles this automatically** — it runs the paste worker on ADD operations. But the agent needs to know:

- Props in the JSON CAN have explicit bindings (template, contentstack, repeater) — the applier will preserve non-static bindings
- Props WITHOUT explicit bindings will get `static_value` bindings from the paste worker
- The `value` field on a bound prop serves as a **fallback** if the binding fails to resolve

**What this means for the agent:**
- You DON'T need to create `static_value` bindings manually — the paste worker handles it
- You DO need to provide `value` fields as defaults/fallbacks
- You DO need to provide non-static bindings explicitly if you want CMS data

---

## 6. Horizontal Stack Auto-Wrapping (DnD Only)

When a user drops a node to the LEFT or RIGHT of another node, the UI auto-wraps both in an `hstack`. **This only happens during DnD, not in JSON generation.**

If the agent wants a horizontal layout, it must explicitly use `flex` with `direction: "row"` or `columns` component. Don't rely on auto-wrapping.

---

## 7. UID Generation Rules

Every node and slot needs a unique UID. The UI uses `nanoid(ID_LENGTH)`.

**Agent rules:**
- Generate unique UIDs for EVERY node: `"hero-section-1"`, `"card-grid-2"`, etc.
- Generate unique slot UIDs for EVERY slot: `"slot-hero-children"`, `"slot-card-content"`, etc.
- NEVER reuse UIDs from existing nodes in the composition
- On MOVE operations, UIDs stay the same (don't generate new ones)
- On ADD operations with copies, use NEW UIDs (the UI calls `rotateAllUids`)

---

## 8. UID Rotation on Duplicate/Copy

When the UI duplicates a node, it calls `rotateAllUids()` which:
1. Replaces the node's UID with a new one
2. For each slot prop, replaces the slot UID with a new one
3. Updates the `slots` object keys to match new slot UIDs
4. Updates `metadata.slotNames` keys
5. Recursively does the same for all children
6. Rotates UIDs inside `json_rte` prop values

**The agent must generate new UIDs for every node when duplicating.** Never reuse the original UIDs — the composition will break with duplicate UID conflicts.

---

## 9. Repeater Wrapper Metadata

When a repeater is inside a container, the parent gets `metadata.repeaterWrapper: true`. This metadata is used by the UI to adjust rendering and DnD behavior.

**The agent should set this when placing repeaters inside wrapper containers.**

---

## 10. Symbol Prop Elevation

When a component is "symbolized" (turned into a reusable symbol), the UI:
1. Extracts all non-symbol_props bindings from child nodes
2. Elevates them to the symbol root as `symbol_props`
3. Remaps child bindings to `{ type: "symbol_props", value: "propKeyName" }`

**If the agent creates symbols, it must do this prop elevation manually.**

---

## 11. Node Normalization

After changes are applied, the editor runs `Editor.normalize()` which auto-fixes certain structural issues. BUT the agent should NOT rely on normalization to fix bad JSON — normalization handles edge cases, not fundamental structure errors.

**Always provide valid node structures. Normalization is a safety net, not a crutch.**

---

## Quick Checklist Before Generating JSON

- [ ] Every node has a unique `uid`
- [ ] Every slot prop has `"slot": "unique-slot-uid"` pointing to a key in `slots`
- [ ] Every `slots` object has the correct keys matching slot prop UIDs
- [ ] Repeater for modular blocks has condition-blocks (one per block type)
- [ ] Each condition-block has a box inside it
- [ ] Repeaters inside repeaters are wrapped in a box
- [ ] Box wrappers for repeaters have `metadata.repeaterWrapper: true`
- [ ] Non-static bindings (template, contentstack, repeater) are explicit on props
- [ ] Image/file bindings use `.url` suffix on path
- [ ] All `value` fields serve as fallbacks even when bound
- [ ] No duplicate UIDs anywhere in the composition
- [ ] Node types match registered component schemas exactly
