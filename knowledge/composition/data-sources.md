# Data Sources in Compositions

## Data Source Types

Studio compositions can pull data from 7 sources:

| Source | Constant | Purpose | When Available |
|--------|----------|---------|---------------|
| **Linked Template Entry** | `template` | Data from the entry connected to the composition via its content type | Composition linked to a CT with `connected_content_type` |
| **Additional Entry Data** | `contentstack` | Manually pinned entries from any content type | Always |
| **Contentstack Query** | `contentstack_queries` | Query-based data source (CT + filters) | SDK supports `PAGE_DATA_QUERIES` |
| **Component Default Data** | `component_props` | Default prop values from component registration | Always |
| **Repeater Data** | `repeater` | Contextual data when inside a repeater (current iteration item) | Component is child of a repeater |
| Static Value | `static_value` | Internal — auto-generated by paste worker | Internal only |
| Symbol Props | `symbol_props` | Internal — for symbol component bindings | Internal only |

## 1. Linked Template Entry (`template`)

The main page entry connected to the composition.

**How it works:**
- Composition is created as "Linked Composition" with a content type selected
- The preview entry of that CT provides the data
- Studio fetches entry using CDA with auto-generated `include[]` paths
- Data flows: CDA → Studio → `UPDATE_DATA_SOURCE` → SDK

**Binding format:**
```json
{
  "binding": {
    "type": "template",
    "value": {
      "path": { "title": {} }
    }
  }
}
```

**Nested paths:**
```json
{ "path": { "author.name": {} } }           // reference field
{ "path": { "hero_image.url": {} } }         // file/image field
{ "path": { "modular_blocks.0.title": {} } } // modular block field
```

## 2. Additional Entry Data (`contentstack`)

Manually pinned entries from any content type.

**How it works:**
- User adds entries via Page Data tab → "Link Entry"
- Each entry stored as data source config
- Studio fetches each with CDA + `include[]`

**Binding format:**
```json
{
  "binding": {
    "type": "contentstack",
    "value": {
      "uid": "blt...",
      "_content_type_uid": "blog_post",
      "path": { "title": {} }
    }
  }
}
```

Required fields: `uid` (starts with "blt"), `_content_type_uid`, `path`

## 3. Contentstack Query (`contentstack_queries`)

Query-based data — user configures a query (content type + filters) and results are fetched.

**How it works:**
- User configures query in Page Data tab
- Studio executes query against CDA
- Results available as data source
- `include[]` is user-configured in query spec, NOT auto-detected

## 4. Component Default Data (`component_props`)

Default prop values defined when registering components.

**How it works:**
- Props defined in `registerComponents()` have default values
- SDK sends component props to Studio
- Available as fallback when no other data source provides a value

## 5. Repeater Data (`repeater`)

Contextual data when a component is inside a repeater iterating over an array.

**How it works:**
- Parent repeater iterates over array data (e.g., modular blocks)
- Each iteration provides current item context
- Child components bind to repeater context fields
- Binding uses `repeaterUID` matching the parent repeater

**Binding format:**
```json
{
  "binding": {
    "type": "repeater",
    "value": {
      "repeaterUID": "repeater-node-uid",
      "path": { "product_name": {} }
    }
  }
}
```

See `repeaters.md` for full repeater patterns.

## include[] Path Generation

When binding to template or contentstack entries, Studio auto-generates `include[]` paths to resolve referenced entries:

```
Binding path: "author.name" (where "author" is a reference field)
→ include[] = ["author"]

Binding path: "modular_blocks.hero_section.hero_ref.bio"
→ include[] = ["modular_blocks.hero_section.hero_ref"]
```

The `include[]` ensures referenced entries are fetched along with the main entry.

## Spec Data Structure (what SDK receives)

```json
{
  "dataSources": {
    "component_props": { /* nested component data */ },
    "template": { /* connected entry data with resolved includes */ },
    "contentstack": {
      "content_type_uid": {
        "entry_uid": { /* resolved entry data */ }
      }
    },
    "static_value": {
      "key_id": "Static value here"
    }
  }
}
```

## Agent Rules for Data Sources

1. When user asks to bind a prop → check what data sources are available in the page data context
2. If template entry exists → prefer `template` binding for main page fields
3. If user references a specific entry → use `contentstack` binding with uid and content_type_uid
4. If data should repeat (list of items) → use repeater with `template` binding on items prop
5. If no CMS data available → use `staticString`
6. ALWAYS verify the field path exists in the page data context before binding
7. ALWAYS append `.url` to image/file field paths
